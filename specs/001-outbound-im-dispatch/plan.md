# Implementation Plan: Outbound IM Dispatch

**Branch**: `feat/outbound-im-dispatch` | **Date**: 2026-03-22 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-outbound-im-dispatch/spec.md`

## Summary

实现 AgentStudio 到 IM（企业微信）的出站消息分发与回复路由功能。用户可在 Web UI 将 AI 生成的消息一键转发到企微群，转发消息携带唯一路由标识 `[#ob_xxxxxx project_name]`；群内用户引用回复后，回复内容自动注入回原始 Agent 会话，形成跨平台双向对话闭环。

技术方案：as-dispatch 新增 `POST /api/im/send` 端点（生成 short_id → 注入路由头 → fly-pigeon 发送 → 保存 context），回调处理增加 `ob_` 前缀 short_id 解析与 AgentStudio inject 注入；AgentStudio 新增 `POST /api/agui/dispatch-im` 代理端点 + 前端转发按钮。

## Technical Context

**Language/Version**: Python 3.11+ (as-dispatch), TypeScript/Node.js (AgentStudio)
**Primary Dependencies**: FastAPI, SQLAlchemy (async), fly-pigeon, React 19
**Storage**: MySQL (production), SQLite (dev) — 复用已有 `outbound_message_contexts` 表，零 schema 变更
**Testing**: pytest (as-dispatch), vitest (AgentStudio)
**Target Platform**: Linux server (as-dispatch), macOS/Linux (AgentStudio dev)
**Project Type**: Cross-repo (as-dispatch + AgentStudio)
**Performance Goals**: 消息分发 < 3s (SC-001)，回复匹配 < 5s (SC-002)
**Constraints**: 不修改现有 HITL-MCP 流程 (FR-010)，fly-pigeon 同步 API 需要 run_in_executor 包装
**Scale/Scope**: 2 个子项目，~6 个文件变更 (as-dispatch)，~4 个文件新增/变更 (AgentStudio)

## Implementation Phases

### Phase A: as-dispatch — 出站发送端点 (P1-Story1 backend)

**Goal**: 实现 `POST /api/im/send`，支持生成 short_id、注入路由头、发送消息、保存上下文。

**Files**:
1. `forward_service/utils/short_id.py` — **NEW**: `generate_outbound_short_id()` 函数
2. `forward_service/routes/im_send.py` — **NEW**: `/api/im/send` 路由
3. `forward_service/routes/__init__.py` — **MODIFY**: 注册 `im_send_router`
4. `forward_service/utils/content.py` — **MODIFY**: 扩展 `SHORT_ID_PATTERN` 支持 `ob_` 前缀

**Key design decisions**:
- `generate_outbound_short_id()`: `"ob_" + secrets.token_hex(3)` → 8 字符
- 路由头格式: `[#ob_xxxxxx ProjectName]\n\n<content>`
- 发送失败不保存 context（保持 context 只有成功发送的记录）
- `fly-pigeon` 的 `send_to_wecom()` 是同步函数，用 `run_in_executor` 包装为 async
- `expires_at` 在创建 context 时显式传入 7 天后时间

**Dependencies**: None（独立模块，使用已有 sender.py 和 repository.py）

### Phase B: as-dispatch — 回调路由增强 (P1-Story2 backend)

**Goal**: 回调处理中增加 `ob_` 前缀 short_id 解析，匹配 outbound context 后注入 AgentStudio 会话。

**Files**:
1. `forward_service/routes/callback.py` — **MODIFY**: 在引用回复解析逻辑中增加 ob_ short_id 分支
2. `forward_service/utils/content.py` — **ALREADY MODIFIED** in Phase A

**Key design decisions**:
- 路由优先级（在现有逻辑基础上调整）:
  1. **NEW**: 从 `quoted_short_id` 判断是否为 `ob_` 前缀 → 查 outbound context → 注入 AgentStudio inject
  2. **EXISTING**: `quoted_message_id` → 查 outbound context（原有逻辑，保持不变）
  3. **EXISTING**: `quoted_short_id` → 查 HITL session（原有逻辑，保持不变）
  4. **EXISTING**: active session fallback

- 注入逻辑:
  - 从 `OutboundMessageContext` 获取 `session_id`
  - 通过 bot 配置中的 `target_url` 推导 AgentStudio 地址
  - 调用 `POST <agentstudio_url>/api/agui/sessions/{sessionId}/inject`
  - 使用 `httpx.AsyncClient` 异步调用（P10 合规）
  - 注入参数: `{ message: reply_text, sender: "wecom-reply", workspace: project_name }`

- 注入失败处理:
  - 记录 ERROR 日志
  - 不影响原有回调流程（fallback 到 HITL 或 active session）
  - 不向企微用户发送错误消息（静默失败）

**Dependencies**: Phase A（需要 `ob_` 前缀 short_id 格式和扩展的正则）

### Phase C: AgentStudio — 后端代理端点 (P1-Story1 backend)

**Goal**: AgentStudio backend 新增 `POST /api/agui/dispatch-im` 代理端点。

**Files**:
1. `agentstudio/backend/src/services/dispatchService.ts` — **NEW**: as-dispatch 客户端
2. `agentstudio/backend/src/routes/agui.ts` — **MODIFY**: 添加 dispatch-im 路由

**Key design decisions**:
- `dispatchService.ts` 封装:
  - JWT token 生成（使用 `jsonwebtoken` 库 + `JWT_SECRET_KEY` 环境变量）
  - `sendToIM(params)`: 调用 as-dispatch `/api/im/send`
  - 超时设置: 10 秒
  - 错误处理: 网络错误 → `{ success: false, error }`

- 环境变量:
  - `AS_DISPATCH_URL`: as-dispatch 地址（如 `http://localhost:8083`）
  - `JWT_SECRET_KEY`: 与 as-dispatch 共享的 JWT 密钥

**Dependencies**: Phase A（as-dispatch `/api/im/send` 必须先实现）

### Phase D: AgentStudio — 前端 UI (P1-Story1 frontend)

**Goal**: 在 AI 消息气泡上添加"转发到企微"按钮和交互。

**Files**:
1. `agentstudio/frontend/src/types/dispatch.ts` — **NEW**: DispatchIM 类型定义
2. `agentstudio/frontend/src/hooks/useDispatchIM.ts` — **NEW**: dispatch API hook
3. `agentstudio/frontend/src/components/chat/DispatchIMDialog.tsx` — **NEW**: 确认弹窗
4. `agentstudio/frontend/src/components/chat/MessageBubble.tsx` — **MODIFY**: 添加转发按钮 + 状态

**Key design decisions**:
- 消息级别状态: `dispatchStatus: 'idle' | 'sending' | 'sent' | 'error'`
- 转发按钮仅在 AI 消息气泡（非用户消息）上显示
- 需要当前项目已配置 `botKey` 和 `chatId` 才显示按钮
- 确认弹窗展示: 目标群信息、消息预览（前 100 字符）
- 成功后气泡显示"已转发"图标 + tooltip（含 short_id）
- 错误时显示 toast notification
- 同一消息可重复转发（每次生成新 short_id）

**Dependencies**: Phase C（需要 `dispatch-im` 端点）

### Phase E: 集成测试与边界处理

**Goal**: 端到端验证完整的消息转发和回复路由流程。

**Scope**:
- as-dispatch: `tests/unit/test_im_send.py`, `tests/unit/test_outbound_routing.py`
- AgentStudio: 手动集成测试（quickstart.md 场景）
- 边界情况验证:
  - 重复转发 → 不同 short_id
  - 过期上下文 → fallback
  - 非引用回复 → 不匹配
  - Bot Key 无效 → 前端错误展示
  - AgentStudio inject 失败 → 静默 fallback

**Dependencies**: Phase A-D 全部完成

### Phase F: P2 — 定时任务/Hook IM 通知 (P2-Story3, deferred)

**Goal**: 支持 scheduled task 和 hook 产出的 AI 响应自动分发到 IM。

**Scope**: 复用 Phase A 的 `/api/im/send` 端点，从 AgentStudio 的 task/hook 编排层调用。

**Status**: Deferred — P1 完成后再规划具体实现。依赖 AgentStudio 的 scheduling/hooks 基础设施。

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| fly-pigeon 同步调用阻塞事件循环 | 高 | 使用 `run_in_executor` 包装（P10 合规方案） |
| 企微引用文本截断导致 short_id 丢失 | 中 | `[#ob_xxx]` 在消息头部（第一行），被截断概率极低 |
| JWT_SECRET_KEY 配置不一致 | 高 | quickstart 文档明确要求两端使用相同密钥 |
| AgentStudio inject 端点超时/不可用 | 中 | 静默 fallback 到 HITL/active session 路由 |
| outbound_short_id 碰撞 | 低 | 6 hex chars = ~16M 空间，7 天窗口内碰撞概率极低 |
