# Feature Specification: 微信个人号多媒体消息收发

**Feature Branch**: `002-weixin-multimedia`  
**Created**: 2026-03-23  
**Status**: Draft  
**Input**: User description: "Extend WeChat personal account (iLinkAI protocol) integration to support bidirectional multimedia messaging — image, voice, file, and video — in addition to existing text-only support."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 接收并处理图片消息 (Priority: P1)

用户通过微信个人号发送一张图片给 Bot。系统能够识别该图片消息，下载并解密图片内容，将图片传递给后端 Agent 进行处理（如图像理解、OCR），而不再显示"暂不支持处理图片消息"的占位文本。

**Why this priority**: 图片是微信中仅次于文本的最高频消息类型。当前所有图片消息被替换为占位符，用户无法向 Agent 分享截图、照片或文档扫描件。这是阻碍多模态交互的首要瓶颈。

**Independent Test**: 可以通过微信发送一张图片给 Bot 并验证 Agent 是否收到图片数据来独立测试——即使不支持发图，Agent 仍可对图片内容进行文字回复。

**Acceptance Scenarios**:

1. **Given** 用户已与 Bot 建立私聊会话, **When** 用户发送一张 JPG/PNG 图片, **Then** 系统解析图片消息，下载解密后将图片数据通过管线转发给 Agent，Agent 收到包含图片信息的 InboundMessage
2. **Given** 用户发送图片, **When** 图片 CDN 下载失败（网络超时或 URL 过期）, **Then** 系统记录错误日志，回退为文本占位提示（如"[图片下载失败]"），不影响后续文本消息的正常接收
3. **Given** 用户发送图片, **When** 图片解密失败（密钥不匹配或数据损坏）, **Then** 系统记录错误日志并返回友好提示，整体消息流不中断

---

### User Story 2 — 发送图片消息给用户 (Priority: P1)

Agent 生成图片（如图表、分析结果可视化、AI 生成图像），系统能够将该图片通过微信发送给用户，用户在微信聊天中直接看到图片，而非仅能看到文本链接。

**Why this priority**: 与 User Story 1 组成完整的图片双向通信闭环。Agent 回复中包含图片（如数据分析图表、设计稿、AI 生图结果）是高频使用场景。

**Independent Test**: 可以向 Agent 发送"画一张猫的图"等指令，验证 Agent 生成的图片是否以微信图片消息形式送达。

**Acceptance Scenarios**:

1. **Given** Agent 处理完用户请求并生成一张图片, **When** 系统通过出站管线发送该图片, **Then** 系统加密图片、上传到 CDN、构建图片消息结构，用户在微信中看到图片消息（而非文本链接）
2. **Given** Agent 回复中同时包含文本和图片, **When** 系统发送混合消息, **Then** 文本和图片按顺序分别发送，用户在微信中同时收到文本和图片
3. **Given** 图片上传到 CDN 失败, **When** 系统检测到上传错误, **Then** 系统回退为发送图片 URL 的文本消息（如果有 URL），或发送错误提示文本，不阻塞后续消息

---

### User Story 3 — 接收语音消息并转为文本 (Priority: P2)

用户通过微信发送语音消息给 Bot。系统能够识别语音消息类型，提取语音内容（优先使用平台内置的文本转写，如不可用则下载语音文件转码后交给 Agent），使 Agent 能理解语音消息的内容。

**Why this priority**: 语音是微信用户的高频交互方式，尤其在移动场景下。支持语音接收让用户可以免打字与 Agent 对话，显著降低使用门槛。

**Independent Test**: 通过微信发送一段语音消息给 Bot，验证 Agent 是否收到了语音对应的文本内容或语音文件数据。

**Acceptance Scenarios**:

1. **Given** 用户发送一段语音消息, **When** 消息的语音项中包含平台转写文本字段, **Then** 系统直接使用该转写文本作为消息内容传递给 Agent，无需额外处理
2. **Given** 用户发送一段语音消息, **When** 平台未提供转写文本, **Then** 系统下载并解密语音文件，将原始语音数据通过管线传递给 Agent（由 Agent 侧决定是否进行语音识别）
3. **Given** 用户发送语音消息, **When** 语音下载或转码失败, **Then** 系统记录错误日志并回退为占位文本提示，不中断消息流

---

### User Story 4 — 接收和发送文件 (Priority: P2)

用户通过微信发送文件（如 PDF、Excel、Word 文档）给 Bot，系统能下载解密文件并转发给 Agent。反之，Agent 生成文件时，系统能将文件通过微信发送给用户。

**Why this priority**: 文件交换是办公协作场景的核心需求。用户需要向 Agent 提交文档进行分析（如简历筛选、合同审核），Agent 也需要返回生成的文件（如报告、表格）。

**Independent Test**: 通过微信发送一个 PDF 文件给 Bot，验证 Agent 是否收到了文件数据和文件名。

**Acceptance Scenarios**:

1. **Given** 用户发送一个文件, **When** 系统解析到文件消息, **Then** 系统下载解密文件，提取文件名和文件内容，转发给 Agent
2. **Given** Agent 生成一个文件需要发送给用户, **When** 系统通过出站管线发送, **Then** 系统加密上传文件到 CDN，构建文件消息结构（包含文件名），用户在微信中收到可下载的文件
3. **Given** 用户发送的文件超出合理大小限制, **When** 系统检测到文件过大, **Then** 系统跳过文件下载并通知用户文件过大，不影响其他消息处理

---

### User Story 5 — 接收和发送视频消息 (Priority: P3)

用户通过微信发送视频给 Bot，或 Agent 生成视频内容后通过微信发送给用户。

**Why this priority**: 视频在消息类型中使用频率最低，且处理复杂度（文件体积大、处理耗时长）最高。作为最后一个阶段实现，是对多媒体能力的完善补充。

**Independent Test**: 通过微信发送一段短视频给 Bot，验证 Agent 是否收到视频数据。

**Acceptance Scenarios**:

1. **Given** 用户发送一段视频消息, **When** 系统解析到视频消息, **Then** 系统下载解密视频，转发给 Agent
2. **Given** Agent 生成视频需要发送给用户, **When** 系统通过出站管线发送, **Then** 系统加密上传视频到 CDN，用户在微信中收到视频消息
3. **Given** 视频文件过大, **When** 下载或上传超时, **Then** 系统记录错误并回退为文本提示，不阻塞其他消息

---

### Edge Cases

- 当用户连续快速发送多张图片时，系统如何保证所有图片都被正确接收处理而不遗漏？
- 当 CDN 下载 URL 中的凭证过期（消息接收延迟导致），系统如何处理？
- 当媒体文件解密后格式无法识别（非预期的图片/音频/视频格式），系统如何处理？
- 当出站图片/文件上传到一半时网络断开，系统如何确保不产生脏数据？
- 当同一条消息中包含多种类型的 item（如文本+图片混合），系统如何处理 item_list 中的多个项？
- 当系统重启后收到延迟的多媒体消息，临时存储的文件已清理，系统如何处理？
- 当 Agent 响应中包含的图片数据格式非标准（如 SVG、WebP），而平台要求特定格式时如何处理？

## Requirements *(mandatory)*

### Functional Requirements

**入站消息解析**

- **FR-001**: 系统 MUST 识别消息项列表中 type=2（图片）、type=3（语音）、type=4（文件）、type=5（视频）的消息项，并按类型分别处理
- **FR-002**: 系统 MUST 从图片、语音、文件、视频消息中提取 CDN 下载所需的加密查询参数和解密密钥
- **FR-003**: 系统 MUST 使用标准对称加密算法（AES-128-ECB + PKCS7 填充）对下载的媒体内容进行解密
- **FR-004**: 系统 MUST 将解密后的媒体文件临时存储，并在使用完毕后进行清理
- **FR-005**: 系统 MUST 为图片消息构建包含图片数据的 InboundMessage（利用已有的 images 字段），使 Agent 可直接获取图片
- **FR-006**: 系统 MUST 为文件消息提取并保留原始文件名
- **FR-007**: 系统 MUST 支持语音消息的优先路径：如果平台提供文本转写，直接使用转写文本；否则传递原始语音数据

**出站消息发送**

- **FR-008**: 系统 MUST 支持发送图片消息——接受图片数据，加密后上传到 CDN，获取媒体引用信息，构建带有图片项的消息结构
- **FR-009**: 系统 MUST 支持发送文件消息——类似图片流程，但包含文件名元数据
- **FR-010**: 系统 MUST 支持发送视频消息——类似图片流程，使用视频对应的媒体类型参数
- **FR-011**: 系统 MUST 通过平台提供的预签名 URL 获取接口获取上传凭证，而非直接访问 CDN
- **FR-012**: 系统 MUST 使用随机生成的 16 字节密钥对出站媒体内容进行加密后上传

**错误隔离与韧性**

- **FR-013**: 多媒体消息处理中的任何失败 MUST NOT 影响文本消息的正常收发（错误隔离原则）
- **FR-014**: 入站媒体下载/解密失败时，系统 MUST 回退为可理解的占位文本提示
- **FR-015**: 出站媒体上传失败时，系统 MUST 尝试回退发送文本替代（如提供文本链接或错误说明），并返回失败结果但不抛出异常
- **FR-016**: 所有媒体操作 MUST 设置合理超时，防止单次操作阻塞整个消息处理流程

**配置与扩展**

- **FR-017**: CDN 基础地址 MUST 可通过配置指定，不能硬编码
- **FR-018**: 系统 MUST 为每种媒体类型的最大允许文件大小提供可配置限制

**兼容性**

- **FR-019**: 所有新增功能 MUST 遵循现有通道适配器接口规范，不得修改基类的抽象方法签名
- **FR-020**: 群聊消息的过滤逻辑 MUST 保持不变——多媒体支持仅限私聊场景

### Key Entities

- **MediaItem（媒体项）**: 消息中的单个媒体条目。包含媒体类型（图片/语音/文件/视频）、CDN 引用信息（加密查询参数）、解密密钥、以及类型特有的元数据（如文件名、语音时长、图片尺寸等）
- **UploadResult（上传结果）**: 媒体上传操作的产物。包含 CDN 引用信息（用于后续消息构建）、上传是否成功的状态、以及错误信息（如有）
- **MultimediaMessage（多媒体消息）**: 统一管线中的消息扩展。在现有 InboundMessage/OutboundMessage 基础上携带媒体附件信息——图片列表、文件引用、语音数据或视频数据

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户发送的图片消息 100% 被系统识别并尝试处理（不再显示"暂不支持"占位文本），成功率不低于 95%（排除网络故障等不可控因素）
- **SC-002**: Agent 生成的图片能在 10 秒内完成加密、上传和发送，用户在微信中直接看到图片而非文本链接
- **SC-003**: 语音消息处理中，当平台提供转写文本时，Agent 在 2 秒内收到文本内容；当需要传递原始语音时，额外耗时不超过 5 秒
- **SC-004**: 多媒体处理中出现的任何错误，不导致同一用户后续文本消息出现超过 1 秒的额外延迟（错误隔离验证）
- **SC-005**: 文件收发支持最常见的办公文件类型（PDF、Word、Excel、PPT、ZIP），文件名在传输过程中完整保留
- **SC-006**: 功能上线后，微信通道的消息类型覆盖率从当前 1 种（纯文本）提升至 5 种（文本、图片、语音、文件、视频）
- **SC-007**: 新增的多媒体功能不引入任何对现有文本消息流程的回归缺陷——全部现有文本消息测试用例通过率保持 100%
